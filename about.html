About this wiki

The Arcane Archive is a comprehensive, text-only fan reference for the co-op wave-survival magic-brawler The Spell Brigade by BoltBlasterGames. It is built and maintained by players.

What this is

A static, search-indexed HTML reference for every spell, hero, artifact, enemy, boss, world, wave, mechanic, curse, modifier, upgrade, and challenge currently shipped in the game. No accounts, no comments, no trackers, no JavaScript build pipeline — just plain HTML, CSS, and one small JS file (js/wiki.js) for progressive enhancements like search and sortable tables.

Sources

All factual content on this wiki is derived from the shipped game files of The Spell Brigade (Unity IL2CPP build). Two tools do the heavy lifting:

  • AssetRipper 1.3.14 — unpacks Unity asset bundles (.bundle, .assets) into readable .prefab, ScriptableObject, and metadata files.
  • Il2CppDumper 6.7.46 — reconstructs the C# class names, enum maps, and method signatures from the IL2CPP-compiled GameAssembly.dll, which is how we know things like StatType, ElementType, RarityType, and the artifact effect class names (e.g. IncreaseDamagePerArmor).

From those two extractions we produced eight JSON data files under data/ (one per entity family), which are the single source of truth this wiki renders from.

Methodology

The wiki was built by a parallel-agent pipeline:

  1. 8 extractor agents ran in parallel, each focused on one entity family (spells, artifacts, characters, enemies, bosses, waves, mechanics, meta). Every extractor verified its output twice against the raw dump before writing JSON.
  2. 8 builder agents consumed those JSON files in parallel to produce the per-family HTML (one builder per family).
  3. 5 reviewer agents spot-checked the rendered HTML against the source data for consistency, correct anchor patterns, cross-link integrity, and accessibility.

The site contract — URL conventions, anchor patterns, template names, cross-linking rules — is documented in data/site-structure.md and was treated as a hard contract by every builder.

Disclaimers

Not affiliated

This wiki is not affiliated with, endorsed by, or sponsored by BoltBlasterGames, the developer of The Spell Brigade. All game names, trademarks, and assets belong to their respective owners. This is a fan project, made for the love of the game.

Text-only

We use no game art, audio, or sprite renderings — not even the logo. This keeps the wiki firmly on the safe side of copyright and makes every page load instantly on any connection. The only visual flourishes are CSS-generated badges and a single unicode rune sigil.

Inferred mechanics

Some game mechanics are inferred from class names and parameter shapes in the dumped IL2CPP code rather than confirmed in-game. These are flagged inline with a callout note block on the relevant page. If you find a discrepancy between what the wiki says and what the game actually does, please assume the game is correct and consider the inference flagged.

Localization keys

The game ships localized strings (item names, descriptions, flavor text) inside compressed Unity addressable bundles that we were not able to fully unpack. Where a name or description has not been resolved, you will see the raw Unity localization key instead — for example SpellName_Arrow, DescriptionEntry_Artifact_07. These will be replaced as soon as we extract the string tables.

Update cadence

This wiki is a snapshot of the game build identified by the hash: 

d121863287924117b1157d5f5d667b78

(as read from boot.config in the shipped game files.) When the developer ships a new build, we re-run the extraction pipeline and regenerate the affected pages. There is no notification system — check the footer of any page for the build hash to see what version of the game you’re looking at.

Found an error?

This is a statically generated site — every page you see was produced from the JSON files under data/ and the templates under templates/. There is no edit-in-browser workflow and no comment section. If you spot a mistake:

  • For a data-level error (wrong stat, missing spell), the fix lives in the corresponding JSON file. After updating the data, the page must be regenerated by re-running the builder pipeline.
  • For a content-level error (typo, broken link, accessibility issue), the fix lives in the HTML template or the page file itself.

See the project README.md for the generation pipeline.
