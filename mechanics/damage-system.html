Damage system

This is the canonical reference for The Spell Brigade’s damage pipeline — what happens between “a spell goes off” and “HP is removed.” The damage calculation guide and every per-spell page links here for the math.

Confidence labels matter on this page

The compiled code dump exposes class layouts and method signatures but not method bodies. Some steps are reconstructed from named constants (e.g., WeakspotDamageMultiplier = 1.5, FriendlyFireMultiplier = 0.25, StatCalculator.Exponent = 0.5714286) and the structure of call sites. Each step below is tagged high, inferred, or guess.

Damage pipeline

From cast to corpse, ordered by when each operation modifies the damage number: 

// 1. Spell evaluates its own Damage stat at cast time            [high]
spell_damage = spellBaseDamage × (1 + Σ %Damage modifiers)

// 2. Build DamageInfo, snapshot crit fields from player stats     [high]
info.Damage         = spell_damage
info.Element        = spell.currentElementType   // from infusion
info.CriticalChance = playerStats.CriticalChance
info.CriticalDamage = playerStats.CriticalDamage

// 3. Hit lands. Spell-specific multipliers (DealsMoreDamageTowardsCenter…)
raw = info.Damage × spellSpecificMultiplier             // [inferred — varies per spell]

// 4. Weakspot hitbox check                                         [high]
raw ×= 1.5  if hitbox.Type == Weakspot

// 5. Critical roll                                                 [high roll, inferred curve]
isCrit = RandomFloat(0, 100) < info.CriticalChance
raw ×= (1 + info.CriticalDamage / 100)  if isCrit

// 6. Element effectiveness (attacker × target)                     [high — signatures + StatTypes 27/30]
raw ×= (1 + attacker.ElementStrengthMultiplier / 100)   // only if Element != None
raw ×= (1 + target.ElementWeaknessMultiplier   / 100)   // only if target carries the stat

// 7. Per-target debuffs & resistances                              [high method names, inferred order]
raw ×= target.GetDamageAfterMultiplier()                 // IDamageMultiplierProvider stack (acid, dark)
raw  = target.CalculateDamageAfterResistances(raw)       // DamageResistance subtract

// 8. Player-only defensive layer                                   [inferred — see armor-dodge.html]
if target is Player:
    if HasDodged(): return 0                              // fires DodgeAttackEventArgs
    raw = max(0, raw − Armor)                             // flat subtract

// 9. Friendly fire & corruption multipliers                        [high — explicit consts]
if attacker is teammate: raw ×= 0.25                      // FriendlyFireMultiplier
if target.IsCorrupted:   raw ×= damageTakenWhileCorruptedMultiplier

// 10. HP removal — store EffectiveDamageDone for popups/analytics  [high]
info.EffectiveDamageDone = raw
target.RemoveHealth(raw, info, isCrit)
Two ordering caveats

Inside HealthContainer.RemoveHealth, the exact order between GetDamageAfterMultiplier, CalculateDamageAfterResistances, CalculateDamageAfterArmor, and the corruption multiplier is not dump-visible. The order above is reconstructed from method-name semantics (resistances are typically applied before flat armor) but isn’t verified. For most builds it doesn’t matter, but mixing big resistance values with high armor can shift outcomes a few percent.

Master formula (one line)

Collapsing the pipeline for the common case of a player spell hitting an enemy: 

final = max(0,
          spellBaseDamage
          × (1 + Σ%Damage)
          × spellSpecificMult
          × (1.5 if weakspot else 1)
          × (1 + CriticalDamage/100) if crit else 1
          × (1 + ElementStrength/100)
          × (1 + ElementWeakness/100)
          × debuffMult
          − targetArmor
        )

PercentAdd vs PercentAddLogarithmic

Damage uses PercentAdd: every +% from items, artifacts, and variants is summed and applied as 1 + Σ%. Three +50% picks become 1 + 1.50 = 2.50×, not 1.5³ = 3.375×. This is the classic Vampire-Survivors-style additive stack: linear, predictable, and easy to reason about.

Two stats deliberately escape that linearity: Range and Size are PercentAddLogarithmic. They use the same additive sum but then raise the 1 + Σ% factor to the StatCalculator.Exponent = 0.5714286 (≈ 4/7) power. 

// Effective multiplier from total stacked %
PercentAdd(100%)             = 2.00×
PercentAddLogarithmic(100%)  = 2.00^0.5714 ≈ 1.487×
PercentAddLogarithmic(300%)  = 4.00^0.5714 ≈ 2.21×
PercentAddLogarithmic(600%)  = 7.00^0.5714 ≈ 3.21×
Why the curve?

Range and Size feed directly into hit-detection geometry — runaway stacking would either make spells unmissable or break the on-screen composition. Damage doesn’t have that problem, so it stacks linearly.

Element effectiveness

ElementEffectivenessCalculator.CalculateEffectiveness is the only place elements directly modify a damage number — the rest of the element system (DOT, chill, ricochet, corruption) happens through separate IElementEffect instances on top of normal damage.

  1. If damageInfo.Element == None, the calculator returns the value unchanged — no elemental scaling.
  2. Otherwise, multiply by 1 + attacker.ElementStrengthMultiplier / 100 (StatType 30, attacker side).
  3. If the target carries it, multiply by 1 + target.ElementWeaknessMultiplier / 100 (StatType 27, target side; typically only players have this).

Both multipliers are additive within themselves (they’re PercentAdd stats), then combined multiplicatively across attacker and target. No element has a built-in advantage against another element — see the matrix.

Critical hits

The roll happens at hit time but the values are snapshotted at cast time:

  1. Spell builds a PlayerSpellDamageInfo and stores the current CriticalChance and CriticalDamage off IPlayerStats.
  2. When the projectile or beam lands, Hitbox.IsCriticalDamage(info) rolls GetRandomFloat(0, 100) and compares against info.CriticalChance.
  3. On a crit, Hitbox.CalculateCriticalDamage returns damage × (1 + CriticalDamage / 100).

See the dedicated Critical hits page for the full snapshot semantics, the Crit Mastery ascension override, and how DamageOverrideData can fake the displayed amount.

Weakspots

Hitboxes typed HitboxType.Weakspot apply a hard-coded ×1.5 multiplier — the const Hitbox.WeakspotDamageMultiplier. It stacks multiplicatively with crit, so a weakspot crit at CriticalDamage = 100 hits for 1.5 × 2.0 = 3.0×.

One-shot damage

OneShotDamage = 999999

The game exposes a 999999 constant used by effects that are supposed to instantly kill a target regardless of HP — Acolyte’s ghost detonation, certain boss-arena kill volumes, the OneHitMode curse’s self-damage path. Anything that uses this number bypasses normal scaling: even with maximum damageTakenWhileCorruptedMultiplier reductions, the post-Armor value remains far above any reasonable HP pool.

Friendly fire

When a player’s damage lands on a teammate (most often via explosions and zone effects), the const FriendlyFireMultiplier = 0.25 on PlayerHealthContainer is applied. This is a flat ×0.25 on the post-element / pre-armor value, then armor and dodge still apply on the teammate’s side. ExplosionDamageInfo also carries separate DamageOnEnemies and DamageOnPlayers fields so a single explosion can deliver different base numbers to each side before that ×0.25 even fires.

Worked examples

Example 1 — Baseline Arrow, no upgrades, normal enemy

A level-1 Arrow with no improvements, no infusion, no crit, hitting a standard zombie body shot. 

// Inputs
spellBaseDamage          = 10          // hypothetical Arrow base
Σ %Damage                = 0
spellSpecificMult        = 1.0
hitbox                   = Default       // not a weakspot
crit                     = false
ElementStrengthMultiplier= 0           // element = None → skipped entirely
ElementWeaknessMultiplier= 0
debuffMult               = 1.0
target Armor             = 0

// Compute
step1: 10 × (1 + 0)              = 10
step3: 10 × 1.0                    = 10
step4: 10 × 1.0                    = 10
step5: no crit                       = 10
step6: element None → skipped        = 10
step7: 10 × 1.0                    = 10
step8: max(0, 10 − 0)              = 10

final = 10

Example 2 — Arrow with 3 epic Damage picks, IncreaseDamagePerArmor at 10 armor, Fire infusion, Ice-weak enemy

Same Arrow base, three epic +25% Damage store picks, IncreaseDamagePerArmor (assume Parameters[0] = 0.03 per armor point, so 10 armor adds +30% Damage dynamically), Fire infusion, ElementStrength +50% from IncreaseElementalDamagePerInfusionPerformed stacks, and the target has ElementWeaknessMultiplier = 25 (it’s an Ice-themed enemy taking +25% elemental damage). Normal body shot, no crit. 

// Inputs
spellBaseDamage          = 10
Σ %Damage                = 0.25 + 0.25 + 0.25 + 0.30  = 1.05   // 3 epic + dynamic from 10 armor
spellSpecificMult        = 1.0
hitbox                   = Default
crit                     = false
ElementStrengthMultiplier= 50                                 // +50% (attacker)
ElementWeaknessMultiplier= 25                                 // +25% (target)
target Armor             = 0

// Compute
step1: 10 × (1 + 1.05)                              = 20.5
step3-5: ×1.0 ×1.0 (no crit)                          = 20.5
step6a: 20.5 × (1 + 0.50)                          = 30.75
step6b: 30.75 × (1 + 0.25)                         = 38.44
step7-8: no debuff, no armor                              = 38.44

final ≈ 38   // 3.84× the baseline Arrow

Example 3 — Crit weakspot with element strength

Same Arrow, three +25% Damage picks, CriticalChance = 100 (guaranteed crit), CriticalDamage = 150 (+150%), Fire infusion, ElementStrengthMultiplier = 20, target is a boss (no ElementWeaknessMultiplier on the target), hits a weakspot for ×1.5. Boss has Armor = 8 (enemy-side defensive scaling — for the example we treat it as a flat subtraction even though enemies usually don’t expose the Armor stat). 

// Inputs
spellBaseDamage   = 10
Σ %Damage         = 0.75      // 3 × 25%
weakspot          = true
crit              = true
CriticalDamage    = 150
ElementStrength   = 20
ElementWeakness   = 0
target Armor      = 8

// Compute
step1: 10 × (1 + 0.75)        = 17.5
step3: × 1.0                  = 17.5
step4: × 1.5 (weakspot)        = 26.25
step5: × (1 + 150/100) = ×2.5 = 65.625
step6: × (1 + 0.20) = ×1.2      = 78.75
step7: × 1.0                  = 78.75
step8: max(0, 78.75 − 8)      = 70.75

final ≈ 71   // 7.1× the baseline Arrow
Order matters — sort of

Multiplicative ordering between crit, element strength, and weakspot doesn’t change the result (multiplication is commutative). What does change the result is whether a buff is additive (lumped into Σ%Damage) or multiplicative (its own factor). Damage picks add into the same bucket; ElementStrength is its own bucket; weakspot is yet another. Build planning is about figuring out which bucket each source actually feeds.

DamageInfo subtypes

Every damage event carries a DamageInfo object. The base class holds the element, the nominal damage, the optional incoming direction, and the final EffectiveDamageDone the popup reads. Subclasses add attribution and special-case routing.

Artifact hooks into the pipeline

Many artifacts attach as computed modifiers that feed into specific pipeline steps. Selected highlights:

Step 1 — Σ %Damage
IncreaseDamagePerArmor, IncreaseDamagePerNearbyPartyMember, IncreaseDamagePerRevivePerformed, DealMoreDamageWhenStandingStill (spell scope).
Step 5 — CriticalDamage bucket
IncreaseCriticalDamagePerRevivePerformed, IncreaseCriticalDamageWhenStandingStill.
Step 6 — ElementStrengthMultiplier
IncreaseElementalDamagePerInfusionPerformed — every infusion you commit during a run pushes this up.
Step 8 — Armor side
AddArmorEqualToPartyLevel, IncreaseArmorPerObjectiveCompleted, BellZoneGivesArmor (variant), ReduceDamageTakenWhenEnemiesInProximity (also armor-routed).

Open questions

Things the dump doesn’t answer
  • Exact ordering between DamageResistance, the damage-multiplier providers, the corruption multiplier, and Armor inside HealthContainer.RemoveHealth.
  • Whether damageMultiplierProviders sum or product-reduce when more than one effect is active on the same target.
  • Whether GetRandomFloat uses [0, 100) or [0, 100] inclusive on the crit roll. The variable comparison strongly implies the half-open form (so 100% chance is still 100%).
  • Whether negative cursed Damage stats clamp to zero before the (1 + Σ) multiplication. No clamping is visible.
