Rarity

Improvement store items and drops in The Spell Brigade carry a tier from the RarityType enum. Each tier has a hard-coded color convention and a pre-baked value, so an “Epic +25% Damage” and an “Epic +18% FireRate” both come from the same roll-up: the store item declares four values (one per rarity), the rarity picker chooses which to apply, and that value becomes a static modifier on the player.

ImprovementValueContainer

Every ImprovementStoreItemData carries one ImprovementValueContainer — a struct with four serialized floats (one per rarity). When you pick the item at a particular rarity, the game calls: 

ImprovementValueContainer.GetValue(rarity) → value

That float becomes the Value of a fresh flat modifier, which is added to the relevant Stat’s modifier list via IPlayerStats.AddCharacterImprovement (or AddSpellImprovement for spell-scoped items).

Pre-baked, not interpolated

The four rarity values are individually authored by the designers — they don’t share a formula like baseValue × rarityMultiplier. That’s why +25% Damage at Epic isn’t always exactly 2.5× the Common value: each tier is hand-tuned per item type.

RandomRarityPicker & Luck

RandomRarityPicker implements ICanPickARarity and is the single source of rarity rolls for stores and drops. It takes the player’s Luck stat (id 9, Flat) as an input and biases the distribution toward higher tiers.

Exact weighting is not dump-visible

The compiled code dump confirms the class name, the ICanPickARarity contract, and the existence of a Luck input — but the actual weight curve (linear? logarithmic? step function?) is in the method body and not exposed. The strong-prior hypothesis is that every extra Luck shifts the cumulative distribution upward toward Epic / Legendary at a diminishing rate, but the exact numbers will need empirical confirmation.

Two stats from the same Stats reference page directly help you control rolls without relying on Luck:

Luck id 9
Bias the per-pick rarity roll upward.
Rerolls id 17
Reroll the current store offerings entirely — a second draw from the same picker.
Bans id 39
Remove a specific item from this run’s store rolls permanently.
Pins id 40
Keep an item across shop visits, freezing its current rarity.

Worked example — upgrade values

Suppose the Damage Up store item is configured with the container: 

ImprovementValueContainer {
    common    = 0.10      // +10% Damage
    rare      = 0.15      // +15% Damage
    epic      = 0.25      // +25% Damage
    legendary = 0.40      // +40% Damage
}

You enter a store with Luck = 5; the picker rolls Epic. The game calls: 

value = ImprovementValueContainer.GetValue(Epic)   // = 0.25
mod   = new flat modifier {
            StatType = Damage,
            Value    = 0.25
        }
IPlayerStats.AddCharacterImprovement(mod)

From that point on, every spell’s Damage stat aggregation includes +0.25 in its sum, contributing +25% via the PercentAdd path. If you take three of these picks across the run, your Damage stat is baseValue × (1 + 0.75) = baseValue × 1.75.

Rarity persists, not magnitude

Once chosen, the flat modifier stores only the chosen float. The fact that it came from an Epic roll is lost — there’s no way for later effects to ask “was this an Epic +25%?” The four-tier system exists only at the moment of rolling.
