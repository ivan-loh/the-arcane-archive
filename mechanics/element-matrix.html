Element matrix

This page is the lookup grid for what packed ElementType you end up with after a sequence of infusions. The full enum has 21 values (None + 5 singles + 5 doubles + 10 dual combos), but only 21 of the 21 × 21 = 441 ordered cells of a hypothetical input-by-input lookup are reachable. The rest are unreachable because: (a) the enum doesn’t define a triple-element combo, and (b) the matrix is symmetric on element pair, so order doesn’t matter once you go dual.

Row = current spell element (before infusion). Column = infusion element you commit (one of the five singles). Cell = the resulting packed ElementType. Cells that the system never produces are marked —.

Reading the diagonal

The diagonal — Fire on Fire, Lightning on Lightning, etc. — always produces the *Double form, which intensifies the corresponding per-element side-effect (stronger DOT, more ricochets, deeper shred, longer chill, faster corruption).

Why the grid stops at five inputs

Three rules collapse the seemingly large 21×21 space:

  1. Infusions are always one of the five singles. You commit Fire, Lightning, Acid, Ice, or Dark — never a Double or Dual.
  2. Once a spell is at Double or Dual, further infusion has no defined target. The enum exposes no triple-element value, and the dump contains no “upgrade dual to dual+” behavior. Practically, the run’s infusion economy caps a spell at two effects.
  3. Dual combo names are sorted by enum id. Acid < Dark < Fire < Ice < Lightning. So Fire-then-Acid and Acid-then-Fire both pack as AcidFire — the matrix is symmetric.

No element-vs-element matrix exists

There is no “Fire is strong vs Ice”

Many genre games ship a fixed matrix that says e.g. Fire deals +50% against Ice. The Spell Brigade does not work this way. Elemental damage is scaled only by the two player-side stats: ElementStrengthMultiplier on the attacker and ElementWeaknessMultiplier on the target. Neither stat distinguishes which element it’s scaling. A target with +50% ElementWeaknessMultiplier takes +50% from every non-None element, not just one.

The only element-pair-specific behavior is that the per-element effect classes are different (FireEffect, IceEffect, etc.). A Fire spell will always burn an Ice enemy because FireEffect is what the element type carries, not because of a matchup table.
