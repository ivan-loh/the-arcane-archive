Critical hits

Crits in The Spell Brigade are governed by two Flat stats — CriticalChance (id 12) and CriticalDamage (id 13). The defining trick: the stat values are snapshotted at cast time (when the spell constructs its PlayerSpellDamageInfo), but the roll happens at hit time on the Hitbox. So mid-flight stat changes don’t affect a projectile that’s already in the air.

Cast-time snapshot

When a spell creates its damage info: 

info = new PlayerSpellDamageInfo(ownerClientId, spellType, playerStats)

// At construction, the constructor reads PlayerStats and copies:
info.CriticalChance = playerStats.CriticalChance.GetValue()
info.CriticalDamage = playerStats.CriticalDamage.GetValue()

From then on, info.CriticalChance and info.CriticalDamage are frozen for the lifetime of that specific hit event — even if the projectile flies for two seconds and you equip a +Crit artifact mid-flight, the in-flight projectile uses the snapshot.

Hit-time roll

On a successful hit, Hitbox.IsCriticalDamage(info) rolls against GetRandomFloat(0, 100): 

Hitbox.IsCriticalDamage(info) {
    return GetRandomFloat(0, 100) < info.CriticalChance
}
Roll scale

The comparison is against CriticalChance on its UI scale (i.e., a player seeing 35% compares against the value 35.0, not 0.35). Whether the upper bound is inclusive or exclusive isn’t dump-visible, but for any realistic non-100 value it doesn’t matter. With CriticalChance >= 100, every hit crits.

Crit damage formula

Hitbox.CalculateCriticalDamage(damage, source) {
    return damage × (1 + source.CriticalDamage / 100)
}

CriticalDamage is an additive percent — at 0 a crit deals ×1.0 (no bonus), at 100 a crit deals ×2.0, at 250 a crit deals ×3.5. Stacking three +50% picks gives CriticalDamage = 150 → crit multiplier ×2.5.

Weakspot stacking

WeakspotDamageMultiplier = 1.5

Hitboxes typed HitboxType.Weakspot apply a hard-coded ×1.5 multiplier from the const Hitbox.WeakspotDamageMultiplier. This stacks multiplicatively with the crit multiplier — they don’t share a bucket. A weakspot crit at CriticalDamage = 150 hits for 1.5 × 2.5 = 3.75× on top of all the normal damage scaling.

Weakspot detection is purely hitbox-driven — the spell has nothing to do with it. A boss’s exposed crystal is a Weakspot hitbox; any projectile that hits it eats the ×1.5. Auras and zone damage that don’t go through a directional hitbox don’t get the bonus.

OneShotDamage

OneShotDamage = 999999

The engine carries a flat 999999 constant used by effects meant to instantly delete a target regardless of HP — the OneHitMode curse’s self-damage path, Acolyte’s ghost-explosion mechanic, and certain boss-arena kill volumes. Anything using this number skips normal scaling considerations. Even at maximum corrupted-damage reduction, the post-Armor value remains far above any reasonable HP pool.

A one-shot is not a crit. The two systems are independent: a one-shot doesn’t need a crit roll, and a crit doesn’t become a one-shot just because it’s big. The constant is a separate sentinel.

DamageOverrideData

After the crit roll, HealthContainer.GetDamageOverrideForDamage takes isCriticalHit as input and can return a DamageOverrideData that overrides the displayed number. This is how the Critical Mastery ascension converts excess CriticalChance (above 100%) into bonus CriticalDamage at hit-resolution time, and how custom interactions can rewrite the damage popup independently of the underlying HP removal.

Worked examples

1. Plain crit, no weakspot

base       = 100
CritChance = 35
CritDamage = 80

roll       = RandomFloat(0,100) = 22.4   // 22.4 < 35 → crit
result     = 100 × (1 + 80/100) = 180

2. Weakspot non-crit

base       = 100
CritChance = 35
CritDamage = 80
weakspot   = true

roll       = RandomFloat(0,100) = 71.3   // 71.3 < 35 false → no crit
result     = 100 × 1.5             = 150

3. Weakspot crit (the spike)

base       = 100
CritChance = 100            // guaranteed crit
CritDamage = 200            // big crit modifier
weakspot   = true

result     = 100 × 1.5 × (1 + 200/100)
           = 100 × 1.5 × 3.0
           = 450          // 4.5× base — and the ×1.5 doesn't pool into +%Damage
Why “crit” and “weakspot” aren’t one bucket

Both effects are multiplicative with their own factor. If they shared an additive bucket (like Damage picks do), ×1.5 + +200% crit would only sum to 3.5× total. Keeping them as separate multiplications gives the 4.5× spike you see in example 3 — small numbers, big results.
