Stat modifiers

Not the same as Covenant modifiers

This page documents the engine’s internal StatModifier class hierarchy — how every +% from every source ends up in a list and gets aggregated. For the run-level Covenant modifiers (Peaceful, Endless, One Less Spell), see Covenant modifiers.

Every numeric change in The Spell Brigade — store picks, artifact effects, curse penalties, variant tweaks — eventually shows up as a StatModifier in the target Stat’s Modifiers list. The class lives at StatModifier (, abstract). Two branches descend: Static (value set once) and Dynamic (value recomputed on every read).

Class hierarchy

StatModifier // abstract, implements IDisposable
├── flat modifier // { StatType, Value } — set once
│ ├── StartingStatModifier // tagged as baseline kit
│ └── HiddenStatModifier // numeric but not shown in UI
└── computed modifier // abstract — GetValue() recomputed on read ├── Artifact effect* // 16 subclasses (see below) └── Spell variant effect* // 6 subclasses

Every concrete modifier overrides two methods:

float GetValue()
The current numeric contribution. Static modifiers return their stored Value; dynamic ones compute live from game state.
StatType GetStatType()
Which stat the modifier feeds into. Decides which Stat’s list it sits in, and therefore whether StatCalculator treats it as Flat, PercentAdd, or PercentAddLogarithmic.
void Dispose() virtual
Dynamic modifiers override this to unsubscribe from game events (artifact mods often subscribe to OnPlayerRevived, OnImprovementAdded, …).

How they aggregate

When something reads a Stat, the aggregation runs every time. Nothing is precomputed, so a dynamic modifier’s changing output is reflected on the very next read. 

// Stat.GetValue() — runs every read
Stat.GetValue() { return StatCalculator.Calculate( baseValue, Modifiers, // List<StatModifier> statCalculationType );
} // PercentAdd path
sum = 0
foreach m in Modifiers: sum += m.GetValue() // dynamic mods recompute here
return baseValue × (1 + sum) // Flat path
return baseValue + Σ m.GetValue() // Logarithmic path
return baseValue × (1 + Σ m.GetValue()) ^ 0.5714286
What-if previews

Stat.GetValueWithExtraModifiers(extras) lets store-UI code compute “what would my Damage be if I bought this?” without mutating the underlying list. Picking the item later just calls AddModifier with the same payload.

Static branch

flat modifier
Stores StatType and a fixed Value. Created by improvement-store purchases, infusion data, and curse stacks. Once added, GetValue() is just a field read.
StartingStatModifier extends Static
Same numeric semantics, but tagged so Stat.GetSumOfNonStartingModifiers() can filter it out. The UI uses this split to show “what you gained this run” separately from baseline kit.
HiddenStatModifier extends Static
Static-numeric modifier the player should never see in any breakdown UI. Used for internal balancing.

Dynamic branch

computed modifier stores a List<float> Parameters (constructor-supplied coefficients) and exposes GetModifierType() returning either Character (feeds PlayerStats) or Spell (feeds a particular equipped spell’s stat block). The two concrete families:

Artifact effect*
Installed when an artifact is equipped. Most subscribe to a game event in their constructor and unsubscribe in Dispose(). GetValue() reads a counter or live game state.
Spell variant effect*
Installed when a spell variant unlocks. Six known subclasses, all reactive to player or context state.

Why some spell-typed mods write Character stats

The split between scope = Spell and scope = Character is about where the modifier is installed, not what it reads. A variant that gives all player damage a +X% buff while inside a Bell zone routes through Character scope so every equipped spell sees it. A variant that only affects its own spell’s Size routes through Spell.

Spell stat lifecycle

Each spell’s per-cast Stat block is built by combining multiple modifier sources, in this order:

  1. Equip — SpellConfiguration.Stats (a list of BaseStat) seeds each Stat with its baseValue and canonical calculation type.
  2. Starting modifiers — SpellConfiguration.StartingModifiers get added as StartingStatModifiers. UI can filter these out.
  3. Store improvements — Each chosen ImprovementStoreItemData picks a per-rarity float from its ImprovementValueContainer and adds a new flat modifier.
  4. Variants — Spell variants register Spell variant effect* instances (dynamic) and/or static tweaks.
  5. Infusions — AddInfusion(SpellType, ElementType) doesn’t add a number; it changes the spell’s outgoing ElementType. Doubling intensifies the element-effect class.
  6. Artifacts — Character-scope Artifact effect*s go on PlayerStats; spell-scope ones go on the matching SpellAttributes.
  7. Curses — Each curse stack pushes (usually negative) static modifiers, or installs hooks like TakeDamageOnLevelUp.
  8. Evaluation — At read time, Stat.GetValue() routes through StatCalculator using the per-type math.
